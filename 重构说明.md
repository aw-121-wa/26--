# 26探险 底盘工程 重构 / 编译修复说明

> 本文件由助手维护，记录对工程做的每一处改动，作为备份与追溯。
> 目标：先把**底盘**跑起来（可编译、可链接出 .elf），上层（地图/任务调度/外设）以后再接。

---

## 0. 背景与范围

- 现状：`26探险` 是对 `travel` 进行的重写版本。重写了上层逻辑（barrier/map/scaner/turn/pid/motor_task 等），但**整套底层 BSP/驱动没进工程、入口没接、构建系统没配**，导致 `build/` 里只有空的 CubeMX 骨架。
- 本轮范围：**只做底盘相关驱动**。明确不纳入：`openmv / rudder / lcd / pic / 蜂鸣器 / LED` 等外设；循线**直接读 GPIO**，**不用 I2C / 灰度**等其它模式。
- 地图/障碍/主任务（`map` / `barrier` / `main_task`）**已纳入编译**——底盘需在地图上测试；本轮仅排除外设与 I2C。

### 本轮纳入编译的模块
- `Motor/`：motor、encoder
- `Math/`：pid、filter（IMU 滤波用）、sin_generate
- `Sensor/`：imu、scaner（仅 GPIO 直读）、bsp_linefollower（前红外，挡板检测）
- `Task/`：motor_task、turn、task_create、temporary_task、main_task
- `App/`：chassis（chassis_api）、map（map.c、map_message.c）、barrier
- `Driver/`：delay、uart、debug_uart、speed_ctrl、sys.h、config.h
- `Core/`：CubeMX 生成部分

> 注：地图 / 障碍 / 主任务（map、barrier、main_task）**纳入编译**——底盘要在地图上测试。
> 仍然排除的是**外设与 I2C**：openmv / lcd / rudder / pic / 蜂鸣器 / LED / gray / IIC。

### 已删除（外设 / I2C，本工程用不到）
- `Bsp/` 整个目录（openmv、lcd、pic、rudder_control、bsp_buzzer、bsp_led）
- `Driver/IIC.*`、`Driver/gray.*`（循线改为直接读 GPIO）

### 1.5 构建配置
- `CMakeLists.txt`：GLOB 纳入 `App/Math/Motor/Sensor/Task/Driver` 的 `.c`，并加入对应 include 目录；`list(REMOVE_ITEM ...)` 移除 `Driver/uart.c`（旧 ALIENTEK 串口驱动，与 CubeMX 的 `USART1_IRQHandler` 冲突且当前未用）。

### 1.7 运行修复（第二轮：让底盘能在地图上跑）

为“一次性跑通”，把重写时丢失/接错的控制环按原工程（travel）的成熟逻辑修好：

1. ✅ **角度补偿已接回**：`turn.c` 的 `getAngleZ()` 改为返回 `归一化(imu.yaw + imu.compensateZ)`，`mpuZreset()` 的基准归零现在真正生效。
2. ✅ **turn.c 恢复 PID 控制**：`Turn_Angle / Stage_turn_Angle / runWithAngle / Turn360Step` 按原工程的位置式 PID 重写（基于 `gyroT_pid/gyroG_pid`，限幅 `motor_all.GyroT/G_speedMax`），不再是开环 ±15 占位。
3. ✅ **chassis_api 接线修复（关键 bug）**：原 `chassis_gyro_angle` 被写入但**从没人读**（motor_task 读的是 `angle.AngleG`）→ 改为统一用共享的 `angle.AngleG`；`Chassis_MotorControl`/`Chassis_DriveDistance_Blocking` 现在正确设置 motor_task 真正消费的量：循线 `Cspeed`、陀螺仪 `Gspeed`+`AngleG`、转弯 `AngleT`。修复前上层命令（坡道/桥上的航向与速度）根本传不到控制环。
4. ✅ **补齐漏配的循迹灯输入引脚**：对照 travel 逐脚核对（见 §1.8），发现新工程 `.ioc`/`gpio.c` **漏配了 5 个循迹脚 PE14 / PC2 / PC3 / PC14 / PC15**（其余 11 脚已配）。新增 `scaner_gpio_init()`（`scaner.c`，全部 INPUT/NOPULL，与 travel 一致），在 `user_init` 中调用。放在驱动里，CubeMX 重新生成也不会丢。**建议你也在 .ioc 里把这 5 个脚设为 GPIO_Input 以保持一致。**

### 1.8 运行修复（第三轮：参考 xunbao 重建导航逻辑）

根因分析：对比 travel 和 xunbao，发现重写版 `Cross()` 有三个架构级缺陷导致"不减速/修正不明显/冲出桥面"：

1. ✅ **新增游龙防护（anti-snake）**：`Chassis_Periodic_Update_5ms()`（`chassis_api.c`，由 `motor_task` 每 5ms 调用）检测 `Scaner.detail & 0xFC3F`（偏移过大）→ 减速到一半 + 强化循线 PID `kp=12/kd=200`，回正后自动恢复原速。这是 xunbao 车"高速不甩尾"的核心机制。
2. ✅ **`Cross()` 改用 chassis API**：所有运动改为通过 `Chassis_SetTargetSpeed` / `Chassis_SetMode` 控制（不再直接写 `motor_all.Cspeed`），速度变化经过 smooth ramping。
3. ✅ **进慢段/障碍前预减速**：70% 处新增判断——如果直行通过（不转弯）且下一段更慢（`nextNode.speed < nowNode.speed`），提前降到下一段速度。解决了"进桥前不减速"问题。参考 xunbao 架构，不使用 travel 的 `change_speed` 查表法。
4. ✅ **新增 `Chassis_SetMode`**：统一的模式切换 API（`chassis_api.c`），`Cross()` 通过它切换模式。
5. ✅ **新增丢线保护**：`Chassis_EnableLineLostProtection()` + `Chassis_Periodic_Update_5ms` 中的检测，连续丢线超 1 秒自动刹车。
6. ✅ **修复 anti-snake PID 参数不恢复的 bug**：`Chassis_Periodic_Update_5ms` 中 anti-snake 覆盖 `line_pid_param.kp/kd` 为 12/200 后，原本不恢复 → 下一段循线用错参数导致振荡。修复：`Chassis_State` 增加 `saved_line_kp/kd` 备份字段，anti-snake 首次命中时备份、清除时恢复；`Chassis_SetMode` 在任何模式切换时检查并恢复被 anti-snake 覆盖的参数。

### 1.9 审查修复（第四轮：致命 bug + 重要问题）

1. 🔴 **修复 `line_weight` 全零（致命 bug）**：`line_weight[16]` 在 travel 中由 `Gray_Init()` 从 `line_weight_default` 复制，但本工程删了 `gray.c` → 权重全 0 → 循线误差恒为 0 → 不循线。修复：新增 `scaner_init()` 做 `memcpy(line_weight, line_weight_default, ...)`，在 `user_init` 中调用。
2. ✅ **修复 anti-snake `saved_line_kp/kd` 初始为 0 导致 restore 设 kp=0**：anti-snake 开启后如果车没偏离（count 从 0 衰减到负 → 清除），restore 会把 kp/kd 设成 0。修复：`anti_snake_restore` 加 guard `saved_line_kp > 0` 才恢复。
3. ✅ **修复 `Cross()` static 变量不被 `mapInit()` 重置**：`route_state`/`is_near_end`/`detect_started` 提取为文件级变量，新增 `Cross_reset()` 由 `mapInit()` 调用，确保第二轮从干净状态开始。
4. ✅ **修复 `Chassis_MotorControl`/`CarBrake`/`Chassis_DriveDistance_Blocking`/`Chassis_Turn_By_StopGyro_Blocking`/`RampCtrl_Blocking` 绕过 `Chassis_SetMode`**：全部改为走 `Chassis_SetMode`，确保 anti-snake 的 PID 恢复在任何模式切换时都触发。`chassis_api.c` 中唯一保留的 `pid_mode_switch` 调用在 `Chassis_SetMode` 内部。

### 1.10 引脚/映射核对结果（已对照 travel，travel 为准）

逐项比对，**驱动层映射与 travel 完全一致**：
- 电机 PWM（`motor.c`）：L0=TIM8.4/3、L1=TIM9.1/2、R0=TIM4.4/3、R1=TIM4.1/2 ✅ 一致
- 编码器（`encoder.c`）：L0=TIM1、L1=TIM2、R0=TIM3、R1=TIM5；方向 `dirct={+,+,-,-}` ✅ 一致
- 循迹 16 路 GPIO（`read_gpio_16ch`）：E14/B8/B7/B6/B5/B4/D7/C15/D5/D4/D3/D1/D0/C3/C2/C14 ✅ 一致
- 前红外 `Infrared_ahead` = PD8 ✅ 一致；IMU = USART3+DMA ✅ 一致
- ⚠️ 唯一差异：上述 5 个循迹脚 CubeMX 未配 → 已用 `scaner_gpio_init()` 补上（见 §1.7-4）。
- 编码器补偿系数 `R0_ENCODER_SCALE=1.08 / R1_ENCODER_SCALE=1.68` 属调参（非引脚），保持现值，需上车实测。

### 1.11 仍依赖硬件 / 上车需自检
- **启动会卡在等挡板**：`zhunbei()` 先 `while(Infrared_ahead)` 等挡板放上再拿走才出发（比赛起跑逻辑）。台架空跑时需手动遮挡 PD8 触发。
- **PID 参数**：`pid_init()` 里电机/循线/转弯/陀螺仪的 kp/ki/kd 是重写里的值，需上车实测微调。
- **printf 未重定向**：`debug_uart` 的 `printf` 当前无 `fputc/_write` 到串口，输出被丢弃；要看日志需先接调试串口（并清理 `uart.c` 与 `it.c` 的 `USART1_IRQHandler` 冲突）。
- **改了 CubeMX 生成文件**（`it.c` 删了 `USART3_IRQHandler`）：日后用 CubeMX 重新生成会被覆盖，需再删一次，或把 IMU 解析挪进 `it.c` 的 `USER CODE` 段。
- **死代码**：`scaner.c` 的递推平均滤波（`value_calculation/Update_line_data/...`）无人调用，本轮未动。
